# Контракт интеграции с WDTT Control Panel

Этот документ описывает opt-in интеграцию с `wdtt-control-panel` версии `0.10.12` или новее. Агент включается оператором явно и не создаёт входящий административный API.

## Локальный интерфейс агента

Агент получает центральную команду, проверяет её и вызывает локальный интерфейс только из этого списка:

```text
createUser(input)
updateUser(sourceUserId, patch, expectedRevision)
deleteUser(sourceUserId)
readUser(sourceUserId)
readNodeSnapshot(cursor)
```

Интерфейс не принимает строки shell, имена процессов, пути, raw JSON Xray, WARP-профили или произвольные поля.

### Допустимые поля пользователя

Центр передаёт только:

- `sourceUserId` — непрозрачный локальный идентификатор;
- `label` — пользовательская метка;
- `expiresAt` — ISO-время окончания доступа либо `null`;
- `enabled` — включён ли доступ;
- `vkHashes` — от одного до четырёх VK-хешей, необходимых WDTT;
- `ports` — три порта WDTT в формате `DTLS,WG,TUN`;
- `expectedRevision` для изменения существующего пользователя.

Точный маппинг на существующую модель WDTT документируется рядом с адаптером. Адаптер обязан отклонять неизвестные поля и сохранять существующие локальные правила авторизации/валидации.

## Снимок узла

`readNodeSnapshot` возвращает массив пользователей в следующей нормализованной форме:

```json
{
  "protocolVersion": "wdtt-fleet/v1",
  "capturedAt": "2026-06-24T12:00:00.000Z",
  "users": [{
    "sourceUserId": "local-7",
    "label": "team-a",
    "enabled": true,
    "expiresAt": null,
    "traffic": { "receivedBytes": 120, "sentBytes": 45 },
    "online": true,
    "devices": [{ "sourceDeviceId": "phone", "label": "Телефон" }],
    "revision": "7",
    "vkHashes": "hash-one,hash-two",
    "ports": "56000,56001,9000"
  }]
}
```

`online` строится по свежему handshake WireGuard на самом узле. Снимок не содержит private keys, публичные ключи устройств, `passwords.json`, токены, конфигурационные файлы, логи, host metrics, Xray или WARP.

## Обязанности агента

1. Подключаться к центру только исходящим соединением.
2. Регистрироваться одноразовым грантом и сверять целевой `node_id` до выполнения действия.
3. Передавать `wdtt-fleet/v1`, версию агента и версию локального адаптера в heartbeat.
4. Сохранять ID команды и окончательную квитанцию до ответа центру. Повтор одной команды возвращает исходный результат.
5. Проверять срок команды, её тип и схему до вызова адаптера.
6. Сериализовать конфликтующие операции для одного пользователя.
7. Отправлять только нормализованный снимок ограниченного размера и короткие коды ошибок.

Агент не открывает входящий административный API. Для текущего протокола используются одноразовый грант и временный узловой Bearer-токен; mTLS — рекомендуемый следующий уровень защиты для production-развёртываний с повышенными требованиями.

## Совместимость и миграция

1. Локальный адаптер и агент выключены по умолчанию; поведение существующей WDTT-панели не меняется.
2. Сначала включается только `readNodeSnapshot` на тестовом узле.
3. После сверки снимков и версий включаются `user.read`, затем изменяющие команды через отдельный feature flag узла.
4. Центр поддерживает текущую и предыдущую major-версию протокола; несовместимый узел получает понятный статус «нужно обновление».
5. Миграции используют expand/contract: сначала добавление полей и dual-read/write, затем backfill и лишь после этого ограничения.

Xray, WARP и полный доступ к серверу остаются вне интеграции навсегда.
